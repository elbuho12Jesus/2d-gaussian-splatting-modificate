view.py + SIBR_remoteGaussian: ¿qué es y qué puedes hacer?

Tu comando python view.py -s ../360_extra_scenes/flowers -m output/m360/flowers_beta_run2 levanta un servidor de render remoto. La app SIBR es solo el cliente que te enseña los píxeles. Aquí qué hace cada pieza, el protocolo por debajo, los modos de visualización ocultos, y los dos errores fáciles de cometer.

Lo primero, deshacemos la confusión. No estabas viendo un mesh. SIBR_remoteGaussian_app no carga geometría (ni .ply de malla ni puntos): se conecta por un socket TCP a tu proceso de Python y, en cada frame, le manda una cámara y recibe de vuelta una imagen ya renderizada (un bitmap de píxeles). Toda la rasterización de las gaussianas pasa en la GPU del lado de view.py. La app solo dibuja el JPEG/raw que le llega. Por eso se llama remote: el render es remoto.

1. La arquitectura: cliente tonto, servidor pesado

Hay dos programas distintos hablando por un cable:

Implicación directa de esto, y la causa de tu cuelgue: como la GPU que renderiza es la del lado de view.py, si lo lanzas en tu PC local, cada movimiento dispara un render de millones de surfels en tu RTX 3080 de 12 GB → OOM. Quieto no pasa nada (no se pide frame nuevo); al mover, revienta. El detalle del porqué está en el diagnóstico de la sesión y en docs/clamp_radio_y_techo_rasterizer.html.

2. Qué hace view.py, línea a línea

El cuerpo es minúsculo (view.py:9-34). Carga el modelo una vez y entra en un bucle infinito:

gaussians = GaussianModel(dataset.sh_degree)
scene = Scene(dataset, gaussians, load_iteration=iteration, shuffle=False)   # carga el .ply del checkpoint
background = torch.tensor([0,0,0] o [1,1,1])

while True:
  if network_gui.conn == None:
      network_gui.try_connect(dataset.render_items)     # espera a que la app SIBR se conecte
  while network_gui.conn != None:
      custom_cam, _, _, scaling_modifer, render_mode = network_gui.receive()   # ← llega la cámara
      if custom_cam != None:
          render_pkg = render(custom_cam, gaussians, pipe, background, scaling_modifer)   # ← rasteriza
          net_image  = render_net_image(render_pkg, render_items, render_mode, custom_cam)
          net_image_bytes = (clamp(net_image)*255).byte()...cpu().numpy()
      metrics = {"#": nº de gaussianas}
      network_gui.send(net_image_bytes, source_path, metrics)   # → manda los píxeles + métricas
Carga (1 vez): Scene(...) lee point_cloud/iteration_XXXXX/point_cloud.ply del -m y las cámaras del dataset (-s). Por eso necesitas los dos argumentos.
Bucle (cada frame): recibir cámara → render() → empaquetar bytes → enviar. Una vuelta por cada imagen que ves. Sin cámara nueva, no hay render.

3. El protocolo por debajo (network_gui.py)

FunciónQué hace
init(ip,port)Abre un socket que escucha en ip:port (por defecto 127.0.0.1:6009). view.py:48 lo fija a --ip/--port.
try_connect()Acepta la conexión de la app SIBR y le manda la lista de modos de render (render_items) para que la app pinte el menú desplegable.
receive()Lee un JSON con resolution_x/y, fov_x/y, z_near/far, la matriz de vista y de proyección, scaling_modifier y render_mode. Construye un MiniCam (una cámara "ligera", sin imagen GT). Ojo: invierte signos de columnas 1 y 2 de las matrices — es el cambio de convención de ejes OpenGL↔COLMAP (network_gui.py:86-89).
send()Manda los bytes crudos de la imagen, luego la cadena source_path (la app la usa para verificar que coincide con la escena que abriste) y un JSON de métricas — aquí solo "#" = número de gaussianas vivas.

Si resolution_x/y llegan a 0 (ventana minimizada), receive() devuelve None y ese frame se salta sin renderizar.

4. Los modos de visualización (esto es lo que más te interesa)

Cuando conectas, view.py manda dataset.render_items (arguments/__init__.py:57) y la app SIBR los ofrece en un desplegable. Cambiar de modo solo cambia qué canal del render_pkg se convierte en imagen (render_net_image, utils/image_utils.py). No re-renderiza nada extra, así que es gratis:

ModoQué vesPara qué te sirve aquí
RGBEl render de color normal.Lo que ya veías. Mirar césped/cielo a ojo.
AlphaOpacidad acumulada por píxel (rend_alpha), coloreada con turbo.El mejor para cazar huecos. Donde no llega ningún surfel, alpha→0 (un color frío/oscuro fijo). Un agujero de fondo salta a la vista como mancha de alpha baja aunque el RGB lo disimule con el color de fondo.
NormalNormales de los surfels remapeadas a (n+1)/2 como color RGB.Ver si el suelo es un plano coherente o está "roto" en mil orientaciones.
DepthProfundidad de superficie (surf_depth) en turbo.Confirmar que el cielo está "lejos" y no hay splats de fondo metidos en primer plano.
EdgeMagnitud del gradiente (Sobel) del RGB.Bordes/nitidez del detalle.
CurvatureGradiente del mapa de normales.Rugosidad de superficie; superficies lisas → oscuras.

Además de cambiar de modo, la app te deja tocar el scaling_modifier (un slider): multiplica el tamaño de todos los splats antes de rasterizar (render(..., scaling_modifer), llega a get_covariance(scaling_modifier)). Ponlo <1 y los surfels encogen hasta verse como puntitos → es la forma de "ver la nube" y detectar si los huecos son falta de splats (puntos ausentes) o splats transparentes.

Receta anti-huecos con el visor: pon el modo en Alpha y orbita el césped/cielo. Si aparecen parches de alpha baja → hueco real (faltan surfels o son casi transparentes). Cruza con Depth para ver si lo que hay detrás está a la distancia correcta. Esto es justo el síntoma que persigues en flowers.

5. Tus dos gotchas (importantes)

① Cargaste el checkpoint de 30000, no el final

A diferencia de render.py (que por defecto usa --iteration -1 = el último), view.py tiene hardcodeado --iteration 30000 (view.py:44). Tu run llega a 50000, así que con

python view.py -s ../360_extra_scenes/flowers -m output/m360/flowers_beta_run2

estabas viendo el modelo a iteración 30000 (un estado intermedio, con menos splats y huecos peores de lo que es el resultado final). Para ver el final, añade explícitamente:

python view.py -s ../360_extra_scenes/flowers -m output/m360/flowers_beta_run2 --iteration 50000

② La GPU que sufre es la de view.py

Si lo lanzas en el portátil (12 GB), navegar 6.5M/7.5M surfels = OOM al primer movimiento. La solución no es bajar mesh_res (eso es solo para mallas y no aplica al visor); es poner view.py donde está la VRAM: el servidor de 95 GB. Ver §6.

6. Cómo navegar de verdad: view.py en el servidor + túnel SSH

La app SIBR corre en tu máquina (tiene ventana), pero el render lo hace el servidor. Como el socket por defecto es 127.0.0.1:6009, se reenvía con un túnel SSH:

# 1) En el servidor (dentro del contenedor docker, donde está la GPU de 95 GB):
#    usa una GPU libre para no pelear con el entreno
GPU_ID=1 docker compose run --rm -p 6009:6009 surfel_env \
  python view.py -s Datasets/flowers -m output/m360/flowers_beta_run3 --iteration 50000 --ip 0.0.0.0

# 2) En tu PC: abre el túnel (si el servidor no es accesible directo)
ssh -L 6009:localhost:6009  usuario@servidor

# 3) En tu PC: lanza el cliente apuntando al puerto local reenviado
./SIBR_remoteGaussian_app --ip 127.0.0.1 --port 6009

Así el servidor de 95 GB rasteriza sin OOM (las vistas nuevas con muchas intersecciones irán lentas, pero no se cuelgan), y tu PC solo recibe píxeles. Notas: --ip 0.0.0.0 hace que view.py escuche en todas las interfaces; con el túnel SSH puedes dejar 127.0.0.1 y reenviar igual. Si lanzas con docker compose run recuerda publicar el puerto con -p 6009:6009.
