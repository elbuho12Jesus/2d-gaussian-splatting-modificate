Lógica del backward del rasterizer + diagnóstico de run7 / run8

Dos cosas en un sitio: (1) el diagnóstico con números reales de los dos últimos runs de flowers (run7 cap 7.5M con densify_until 35k, y run8 cap 9M), y (2) una revisión paso a paso del backward CUDA (cuda_rasterizer/backward.cu) porque sospechabas que el error del fondo viene de ahí. Conclusión adelantada: el backward tiene tres divergencias reales respecto al kernel beta que conviene corregir, pero los números de run7/run8 dicen que el grueso del error de fondo NO es el backward — es el techo estructural del surfel 2D (cobertura rasante), ya que subir el cap de 7.5M→9M no mueve el test.

Todo medido, no estimado: PSNR de logs/flowers7.log y logs/flowers8.log (--eval, test honesto); distribución de β de los plys reales iteration_50000 de flowers_beta_run7 (7.5M) y run8 (9M); el código citado es submodules/diff-surfel-rasterization/cuda_rasterizer/{backward.cu, forward.cu} y scene/gaussian_model.py en su estado actual.

Dos lecturas, las dos malas para las hipótesis "más capacidad" y "cortar densify":

  • El cap está saturado para el test. Pasar de 7.5M a 9M splats (+20% de primitivas) sube el test +0.06 dB (21.06→21.12) — ruido. El train sí sube más (24.64→25.14, +0.50 dB): la capacidad extra solo memoriza vistas de train. La brecha train−test se ensancha (3.6→4.0 dB). En flowers, "cap" dejó de ser la palanca dominante: ya no compra generalización.
  • Cortar densify en 35k NO frenó la caída del pico de 30k. Era la apuesta de run7: como en runs 4/5/6 el test picaba ~30k y caía, se cortó relocate/add_new/ruido/regs en 35k esperando que los últimos 15k consolidaran. Pero el test igual pica @30k (21.31) y cae a 21.06 @50k (−0.25) — con densify ya apagado. Conclusión: la caída tardía no la causa la densificación ni el ruido; la causa el ajuste fotométrico puro + lr siguiendo bajando = el modelo sigue memorizando train aunque no añada ni mueva splats por MCMC.

Las curvas de run7 (7.5M, sólidas) y run8 (9M, discontinuas) se solapan en test. La única diferencia visible es que el train de run8 sube más alto = más memorización. La tijera entre verde y rojo tras 30k es el sobreajuste del fondo, idéntico con +20% de capacidad.

Esto repite, ahora con cap 9M, el diagnóstico de flowers4/5: el defecto dominante es generalización del fondo, no falta de primitivas. Es el contexto en el que hay que leer la revisión del backward.

2. El backward de un vistazo — quién calcula qué gradiente

El backward del rasterizer (backward.cu, 787 líneas) tiene tres núcleos que se ejecutan en este orden inverso al forward:

NúcleoGranularidadQué derivaSale hacia
renderCUDA (l.143–495)1 hilo / píxelRe-rasteriza de atrás-a-delante; reparte el gradiente del píxel entre los splats que lo tocarondL_dcolors, dL_dopacity, dL_dbeta, dL_dtransMat, dL_dmean2D, dL_dnormal3D
compute_transmat_aabb (l.498–628)1 hilo / splatDe dL_dtransMat (la matriz 3×3 del surfel) a los parámetros crudosdL_dmeans (posición 3D), dL_dscales, dL_drots
computeColorFromSH (l.20–139)1 hilo / splatDe dL_dcolor a los coef. SH + el trocito de gradiente de posición por la dependencia de vistadL_dshs, += dL_dmeans

La ecuación central que reparte el gradiente en renderCUDA es el alpha-blending de atrás a delante. Para cada par (splat i, píxel):

αi = oi · Gi ,   Gi = (1 − ρ)β  ←  kernel beta (no gaussiano)

donde ρ = min(ρ3d, ρ2d): ρ3d = sₓ²+s_y² es la huella real del surfel proyectado y ρ2d = 2·(dₓ²+d_y²) es el filtro pasa-bajo de 2 px (FilterInvSquare=2). El backward bifurca según cuál de los dos ganó el min (l.444):

  • rama rho3d (ρ3d ≤ ρ2d, splat grande en pantalla): gradiente completo a la matriz transMat → posición/escala/rotación 3D. Es la rama que entrena la geometría de los splats grandes de fondo.
  • rama rho2d (splat sub-píxel): gradiente de posición a dL_dmean2D + solo la profundidad a transMat.

3. La matemática del kernel beta — forward vs. backward

El forward (forward.cu) define el kernel beta y satura α:

// forward.cu l.397-416
float power      = -0.5f * rho;            // ← VESTIGIAL (gaussiano), nunca se usa
float beta_j     = collected_beta[j];      // beta_j = 4·exp(clamp(_beta,-6,2))  ∈ [0.01, 29.6]
float one_minus  = max(1.0f - rho, 1e-6f);
float kernel     = powf(one_minus, beta_j);   // (1-ρ)^β
float alpha      = min(0.99f, opa * kernel);

La activación de get_beta en gaussian_model.py es 4·exp(b) con b∈[−6,2], así que el exponente real β vive en [0.01, 29.6] (init _beta=1 → β≈10.9; mediana medida ≈ 2.8).

Las derivadas correctas del kernel beta son:

∂α/∂β
α · ln(1 − ρ)
implementado bien l.408
∂α/∂ρ
− β · α / (1 − ρ)
implementado bien l.420 (d_alpha_d_rho)

El derivado de la huella respecto a las coords del surfel s que debería usarse es, por la cadena ρ3d = sₓ²+s_y²:

∂G/∂sₓ = −2·β·G/(1−ρ) · sₓ   (beta)   ≠   −G·sₓ  (gaussiano)

4. Hallazgo #1 — la rama rho3d usa la derivada GAUSSIANA bug real

En la rama de los splats grandes (l.444–449), el gradiente a la matriz del surfel se calcula así:

// backward.cu l.439-449
const float dL_dG = nor_o.w * dL_dalpha;        // = o · dL/dα   (∂α/∂G = o, correcto)
...
if (rho3d <= rho2d) {
    const float2 dL_ds = {
        dL_dG * -G * s.x + dL_dz * Tw.x,   // ← ∂G/∂sₓ = -G·sₓ  ES GAUSSIANO
        dL_dG * -G * s.y + dL_dz * Tw.y
    };

El factor -G·s es ∂(exp(−0.5ρ))/∂s — la derivada de un gaussiano, no del kernel beta. Para el kernel beta el factor correcto es −2β·G/(1−ρ)·s. Es el mismo "resto gaussiano" que el power=-0.5*rho vestigial del forward: el port cambió α al kernel beta y arregló ∂α/∂ρ para la rama 2D (l.420), pero dejó la rama 3D con la fórmula vieja.

El error es un factor multiplicativo por par (splat,píxel):

gradcódigo / gradcorrecto = (1 − ρ) / (2β)

Con β mediana ≈ 2.8 y ρ pequeño → factor ≈ 0.18: el gradiente de geometría de los splats grandes sale ~5× demasiado pequeño y con la dependencia de ρ equivocada.

Matiz honesto sobre el impacto: el factor (1−ρ)/(2β) es positivo, así que no invierte la dirección del gradiente. Además es casi-constante por splat → Adam lo absorbe en gran parte (Adam normaliza por la magnitud del gradiente). Lo que Adam NO cancela es la parte 1/(1−ρ), que re-pondera entre píxeles: el kernel beta debería pesar más los píxeles del borde del splat (ρ→1); la fórmula gaussiana los pesa por el valor del kernel, que cae distinto. Resultado práctico: la escala del gradiente la salva Adam, pero la forma (qué píxeles mandan al re-moldear el surfel) está sesgada hacia el centro. Para los surfels grandes y aplanados del fondo, esto sub-entrena su forma/orientación de borde. Es una divergencia de corrección real, candidata a arreglar y medir, pero no es un error catastrófico.

Esquema cualitativo: las dos fórmulas reparten el gradiente de forma de manera distinta a lo ancho del surfel. Adam arregla la altura global de cada curva, pero no que tengan forma distinta.

5. Hallazgo #2 — el continue con β<0.1 congela TODA la geometría bug real, impacto pequeño

// backward.cu l.402-405
// FIX 2: stop beta backprop if beta is too small
if (beta_j < 0.1f) {
    continue;     // ← salta el RESTO del cuerpo del bucle
}

El comentario dice "frenar el backprop de beta", pero el continue está colocado antes de todo el cálculo de geometría y opacidad. Lo que hay después del continue y por tanto se salta:

  • el gradiente de β (l.414) — lo único que se quería saltar;
  • el gradiente de opacidad (l.492, dL_dopacity);
  • todo el gradiente de geometría (transMat / mean2D, l.444–489) → posición, escala, rotación.

(El gradiente de color sí sobrevive porque está antes, l.340–353.) O sea: un splat con β<0.1 visto desde ese píxel solo actualiza su color; su geometría y opacidad quedan congeladas para ese par. β<0.1 ⟺ 4·exp(b)<0.1 ⟺ _beta < −3.69 = kernel casi plano (disco boxy), típico de splats que intentan cubrir fondo ancho.

Es un bug real pero de impacto pequeño — no explica un defecto que afecta a regiones enteras de fondo. (Conviene arreglarlo igual: mover el continue a guardar solo el atomicAdd de β.)

6. Hallazgo #3 — orden del término de fondo vs dL_d_rho divergencia inerte

El término de "fondo que se asoma cuando no queda α por delante" (l.432–435) se suma a dL_dalpha después de calcular dL_d_rho (l.426):

float dL_d_rho = dL_dalpha * d_alpha_d_rho;     // l.426  — SIN el término de fondo todavía
...
dL_dalpha += (-T_final / (1.f - alpha)) * bg_dot_dpixel;   // l.435 — término de fondo
const float dL_dG = nor_o.w * dL_dalpha;        // l.439 — este SÍ lleva el fondo

Es decir, dL_dG (rama 3D) incluye el fondo, pero dL_d_rho (rama 2D) no. En el 2DGS original las dos ramas usan dL_dG (con fondo), así que esto es una divergencia. Pero es inerte: dL_d_rho solo alimenta dL_dmean2D (l.482–483), y dL_dmean2D se sobrescribe entero en el preprocess (ver §7). El término de fondo que falta en dL_d_rho nunca llega a ningún parámetro. No hay que tocarlo con prisa, pero conviene anotarlo para no confundirse al leer el código.

7. El "hack" de dL_dmean2D — por qué la rama 2D no entrena posición

Al final del preprocess (l.682–684) hay esto, heredado del 2DGS oficial:

// preprocessCUDA l.682-684  — "hack the gradient here for densitification"
float depth = transMats[idx * 9 + 8];
dL_dmean2Ds[idx].x = dL_dtransMats[idx*9+2] * depth * 0.5 * W;   // SOBRESCRIBE
dL_dmean2Ds[idx].y = dL_dtransMats[idx*9+5] * depth * 0.5 * H;

dL_dmean2D se recalcula desde la matriz transMat y machaca lo que renderCUDA había acumulado. Consecuencias, importantes para entender el sistema:

  • dL_dmean2D no es el gradiente que mueve el splat — eso lo hace dL_dmean3D vía transMat. dL_dmean2D es solo la señal de densificación (el xyz_gradient_accum / error que usa MCMC, mencionado en CLAUDE.md).
  • Por tanto, en la rama 2D (splats sub-píxel), el gradiente en el plano calculado en l.482–483 se descarta para todo: ni mueve el splat (lo sobrescribe el hack) ni entrena posición 3D (la rama 2D solo manda profundidad a transMat, l.486–488). Esto es así también en el 2DGS oficial — no es un bug introducido aquí, es una propiedad conocida del surfel 2D: los splats sub-píxel apenas reciben gradiente de posición en plano.

Para el problema del fondo, la implicación es: la geometría del fondo se entrena por la rama 3D (splats grandes), que es justo donde está el Hallazgo #1 (derivada gaussiana en vez de beta).

8. Veredicto

Lo que el backward sí tiene mal (a corregir y medir):

  1. #1 rama rho3d con derivada gaussiana (-G·s en vez de -2β·G/(1−ρ)·s). Afecta a la geometría de todos los splats grandes — la población que cubre el fondo. Impacto amortiguado por Adam en escala, pero sesga la forma del gradiente. Es el candidato más serio.
  2. #2 continue con β<0.1 congela geometría+opacidad, no solo β. Bug claro pero solo el 0.10% de splats lo tocan.
  3. #3 orden del término de fondo en dL_d_rho: divergencia inerte (la consume un valor sobrescrito).

Lo que los datos dicen sobre la causa del fondo:

run8 (cap 9M) no mejora el test sobre run7 (+0.06 dB) mientras el train sube +0.50 — el techo no es capacidad ni, casi seguro, el backward. Si el #1 fuera el cuello de botella del fondo, +20% de splats grandes habría movido algo el test; no lo movió. El patrón (pico test @30k, brecha train−test que crece con más capacidad) es el de sobreajuste estructural por cobertura rasante del surfel 2D, ya diagnosticado en flowers4/5. El backward #1 puede estar limando calidad del fondo, pero el grueso del error es el mismo de siempre.

Recomendación de orden

  • Barato y aislado primero: arreglar #1 (cambiar -G*s por -G*s * 2*beta_j/one_minus en l.447–448) y #2 (mover el continue). Es C++ → recompilar el rasterizer (pip install --force-reinstall --no-deps .../diff-surfel-rasterization). Lanzar un run de control idéntico a run7 y mirar si el fondo en test/vis/ mejora y si el pico de test sube. Coste casi nulo, y resuelve la duda "¿era el backward?".
  • Si el control no mueve el test (lo más probable según run8): confirma definitivamente que el techo es estructural → la palanca real es migración a gsplat/3D (#5b) o, más barato, early-stop ~30k (el test SIEMPRE pica ahí; la 2ª mitad solo memoriza — vale para todos los runs 4–8).
